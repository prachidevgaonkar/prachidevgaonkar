### Ambiguity → shipped roadmap

The same skill that builds AI apps also builds AI programs.
Both start with the same question: *what is this actually supposed to do?*

**TPM mode** — make the fuzzy thing concrete before the first sprint.

```
Vague ask  →  "We need an AI agent for our knowledge base"
My output  →  Data source map. Failure modes. Grounding controls.
               System instructions. Eval criteria. Backlog.
What ships →  Something that survives real queries. Not a demo.
```

**PM mode** — define what "working" means before a single line is written.

```
Vague ask  →  "Users don't trust the AI responses"
My output  →  Root cause: hallucination? grounding gap? wrong persona?
               User stories with acceptance criteria tied to real trust signals.
               Metrics: groundedness score, fallback rate, correction rate.
What ships →  Something measurable. Not vibes.
```

| Situation | What I produce |
|-----------|----------------|
| No alignment on "done" | OKRs + success metrics before sprint 1 |
| Engineering needs direction | PRD — user stories, acceptance criteria, edge cases |
| Scope creeping | RAID log, dependency map, milestone gates |
| Pressure to ship early | Release readiness checklist, UAT plan, rollback criteria |
| Launched but no signal | Adoption metrics, feedback loops, iteration roadmap |
